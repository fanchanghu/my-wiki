# 技术方案与架构文档编写器

本技能帮助将零散的技术思路、改进想法或系统架构，转化为结构清晰、可落地的技术文档。

## 适用场景

在以下情况下使用本技能：
- **技术改进/改造方案**：用户提出一个技术改进思路，需要形成正式方案文档
- **功能设计文档**：为某个功能或模块编写设计文档
- **系统架构文档**：描述现有系统的架构原理、组件关系、核心流程、隔离机制等（无变更）
- **技术选型评估**：需要评估多个技术选项并给出决策依据
- **重构/改造影响面梳理**：需要梳理变更范围和影响面

**关键区分**：
- **设计文档**（面向变更）：回答"要改什么、怎么改、影响面多大"
- **架构文档**（面向描述）：回答"系统长什么样、怎么工作、各组件如何协作"

## 工作流程

### 步骤 1：确定文档类型

根据用户意图判断文档类型，选择对应模板：

| 文档类型 | 典型触发词 | 核心目的 |
|---------|-----------|---------|
| **设计文档** | "改进"、"改造"、"方案"、"重构"、"新增功能" | 描述变更内容、决策依据、实现路径 |
| **架构文档** | "架构"、"原理"、"流程"、"机制"、"整体"、"交互" | 描述现有系统的结构、组件关系、运行方式 |

如果用户意图不明确，主动询问："这是一份面向变更的设计文档，还是面向现有系统的架构说明文档？"

### 步骤 2：现状评估与影响面调查

确定文档类型后，有针对性地调查当前实现：

**设计文档侧重**：
1. **定位核心代码**：找到与变更相关的源码文件、头文件和测试文件
2. **调查影响面**：用 grep 搜索关键符号（函数名、常量、宏）在代码库中的使用范围
3. **识别能力缺口**：当前基础设施是否支持方案？如不支持，需明确额外的实现成本

**架构文档侧重**：
1. **探索组件结构**：找到核心模块、trait/接口定义、数据流路径
2. **定位关键流程**：找到创建/初始化/交互等流程的入口函数
3. **提取机制细节**：找到隔离、并发、存储等机制的实现代码

**关键动作**：
- 搜索相关 API 的使用位置和调用方式
- 搜索关键常量/状态字段的引用点
- 评估第三方库或内部工具的能力边界

### 步骤 3：编写大纲并确认

确定文档类型后，**不要直接编写完整文档**。先根据选定的模板，编写文档大纲（章节结构 + 每章核心内容简述）发给用户确认。

- 说明每个章节要写什么、为什么放在这个位置
- 如涉及 Mermaid 图，标注图的类型和主题
- 如涉及代码引用，标注要引用的关键结构体或接口
- **等待用户明确确认后，再进入步骤 4**

> 同样，当用户要求对已有文档进行**大篇幅修改**（如重构章节结构、新增完整章节、重写核心流程）时，也必须先出修改后的大纲，用户认可后再执行改写。

### 步骤 4：编写完整文档并保存

大纲确认后，按模板编写完整文档并保存到 `docs/` 下的合适位置。保存完成后，提醒用户去该位置查看。

如需大篇幅修改，回到步骤 3 重新确认大纲后再改写。

#### 模板 A：设计文档（面向变更）

适用于技术改进、功能设计、重构方案等场景。

```markdown
# {标题}

## 1. 背景
- 当前设计存在的问题
- 触发此次设计的业务/技术动机

## 2. 目标
- 期望达到的效果（1-3条）

## 3. 接口与行为变更
- 旧方案/旧写法（标记为废弃或待改造）
- 新方案/新写法（展示改进后的效果）
- 变更不限于 API，也可能是配置格式、交互协议、数据结构等
- 如有交互流程，提供时序或状态图描述

## 4. 实现思路
- 按模块分小节描述
- 明确哪些能力需要新增、哪些逻辑需要重写
- 对未定事项使用"（待定）"标注

## 5. 关键变更点
- 以表格形式列出受影响的文件/模块和具体变更内容
```

#### 模板 B：架构文档（面向描述）

适用于描述现有系统架构、组件关系、核心流程、隔离机制等场景。**参考 multi-agent-architecture.md 的成功实践**。

```markdown
# {标题}

## 1. 概述
- 系统/模块的定位与核心职责
- 设计目标（如：隔离性、并发性、可扩展性）
- 术语表

## 2. 整体架构
- 核心组件总览：列出每个组件的职责（Agent、Channel、Workspace、Session、Memory、Tools、SecurityPolicy 等）
- 架构分层图（Mermaid）：展示接入层 → 路由层 → 运行时层 → 基础设施层的分层关系
- 组件交互关系图（Mermaid）：展示各组件之间的数据流和依赖关系

## 3. {核心流程 1} 流程
- 创建/初始化/配置等关键流程的详细说明
- 使用 Mermaid 流程图展示步骤流转
- 每个步骤附简要说明

## 4. {核心流程 2} 交互流程
- 用户与系统、组件与组件之间的消息交互
- 使用 Mermaid 时序图展示交互过程
- 标注关键 frame/消息类型

## 5. {机制名称} 机制
- 隔离机制、并发策略、存储策略等深度机制
- 分维度说明（如 Workspace 隔离、记忆隔离、配置隔离）
- 必要时引用核心代码结构（如关键结构体、trait 定义）辅助说明
- 如支持受控共享，单独说明共享边界和安全约束

## 6. {机制名称 2} 策略
- 并发控制、资源调度、负载均衡等策略
- 说明控制模型、上限常量、抢占/取消机制
- 使用 Mermaid 时序图展示并发调度过程

## 7. 总结
- 架构设计要点回顾（3-5 条）
- 未来扩展方向
```

**架构文档特有要求**：
- 每章主标题下必须先给出一段总体描述，说明本章核心内容和上下文关系，再进入子小节
- 架构图、流程图、时序图优先使用 Mermaid 语法
- 允许引用关键代码片段（如结构体定义、trait 接口、核心常量）来辅助说明机制，但避免罗列完整实现逻辑
- 隔离类机制需从多个维度展开，每个维度独立成小节
- 术语表必须在文档开头统一列出

## 文档质量要求

### 必须包含

#### 设计文档
- [ ] 背景：为什么需要这个方案，当前痛点是什么
- [ ] 目标：具体、可衡量的预期结果
- [ ] API/接口变更：新旧对比，让调用方一目了然
- [ ] 实现思路：高层模块划分，不陷入代码细节
- [ ] 关键变更点：文件级影响范围清单

#### 架构文档
- [ ] 概述：系统定位、设计目标、术语表
- [ ] 整体架构：组件清单 + 分层图 + 交互关系图
- [ ] 核心流程：至少一个关键流程的详细说明 + 流程图/时序图
- [ ] 机制详解：至少一个深度机制的多维度说明
- [ ] 总结：设计要点回顾 + 扩展方向

### 应当避免

- 不写代码实现细节，文档应聚焦"做什么"和"为什么"
- 避免纯文字堆砌，用图（Mermaid 架构图/流程图/时序图）辅助说明更佳
- 避免只有高层概述而无具体机制解释
- 避免遗漏隔离边界、并发约束、安全策略等关键约束条件
- 不罗列完整业务逻辑代码，但允许引用关键接口/结构体定义

## 输出格式

使用本技能时，我会：

1. 确定文档类型（设计文档 vs 架构文档）
2. 针对性调查现状代码和影响面
3. 编写大纲并发送给用户确认
4. 大纲确认后，编写完整文档并保存到 `docs/` 下的合适位置
5. 提醒用户去保存位置查看文档
